# 13장 검색어 자동완성 직접 구현 정리

## 목표

13장 검색어 자동완성 시스템에서 설명한 트라이 기반 조회 구조를 Spring Boot 데모로 직접 구현했다. 구현의 초점은 단순히 `prefix%` 검색을 하는 것이 아니라, 한글 입력 중간 상태와 초성 검색까지 처리할 수 있도록 검색어를 여러 색인 키로 변환하고, 트라이 노드마다 인기 검색어 top-K를 캐시하는 것이다.

구현 위치는 `autocomplete-demo`이며, 직접 구현 엔진은 `TrieAutocompleteEngine`이다.

## 전체 구조

```text
브라우저
  -> /api/autocomplete/compare?prefix=한&limit=5
  -> AutocompleteApiController
  -> AutocompleteComparisonService
  -> TrieAutocompleteEngine
  -> AutocompleteTrie
  -> KoreanAutocompleteNormalizer
```

비교를 위해 `korean-utils` 라이브러리 기반 엔진도 함께 두었다. 화면과 API에서는 직접 구현 트라이 결과와 라이브러리 결과를 같이 반환해서, 같은 입력에 대해 두 방식이 어떻게 동작하는지 비교할 수 있다.

## 직접 구현 엔진

`TrieAutocompleteEngine`은 애플리케이션 시작 시 샘플 검색어 목록을 받아 `AutocompleteTrie`에 모두 삽입한다.

```java
private final TrieAutocompleteEngine trieEngine =
        new TrieAutocompleteEngine(terms, normalizer, CACHE_SIZE);
```

`CACHE_SIZE`는 10으로 두었고, 각 트라이 노드에는 해당 접두어에서 바로 반환할 수 있는 인기 후보를 최대 10개까지 저장한다. API 요청의 `limit`은 1부터 10 사이로 보정해서 과도한 요청을 막는다.

## 한글 검색 키 정규화

한글 자동완성은 완성형 음절 그대로 비교하면 입력 중간 상태를 처리하기 어렵다. 예를 들어 사용자가 `ㅎ`, `하`, `한`처럼 조합 중인 상태로 입력할 수 있기 때문이다.

그래서 `KoreanAutocompleteNormalizer`에서 검색어를 다음 키들로 변환한다.

| 원문 | 생성 키 예시 | 목적 |
| --- | --- | --- |
| `한국 여행` | `ㅎㅏㄴㄱㅜㄱ ㅇㅕㅎㅐㅇ` | 실제 입력 순서에 가까운 자모 검색 |
| `한국 여행` | `ㅎㅏㄴㄱㅜㄱㅇㅕㅎㅐㅇ` | 공백을 생략한 입력 처리 |
| `한국 여행` | `ㅎㄱㅇㅎ` | 초성 검색 처리 |

정규화 방식은 다음과 같다.

- 한글 완성형 음절은 유니코드 값으로 초성, 중성, 종성 인덱스를 계산해 자모 문자열로 분해한다.
- 복합 모음과 복합 받침은 입력 순서에 가까운 자모 조합으로 풀어낸다. 예를 들어 `과`는 `ㄱㅗㅏ`, `값`은 `ㄱㅏㅂㅅ`이 된다.
- 공백은 일반 검색 키에서는 하나로 정리하고, compact 검색 키에서는 제거한다.
- 영문은 소문자로 통일한다.
- 이미 자모로 들어온 입력도 가능한 범위에서 같은 키 체계로 맞춘다.

## 색인 흐름

검색어 하나를 삽입할 때는 `normalizer.indexKeys(term.query())`로 여러 키를 만든 뒤, 각 키를 트라이에 삽입한다.

```text
SearchTerm("한국 여행", 15420)
  -> ㅎㅏㄴㄱㅜㄱ ㅇㅕㅎㅐㅇ
  -> ㅎㅏㄴㄱㅜㄱㅇㅕㅎㅐㅇ
  -> ㅎㄱㅇㅎ
  -> 각 키를 트라이에 삽입
```

삽입 과정에서는 루트부터 각 문자 노드를 따라 내려가며 모든 경로 노드에 현재 검색어를 `remember`한다. 이때 노드의 `topSuggestions`는 항상 빈도 내림차순, 검색어 오름차순으로 정렬되고, 캐시 크기를 넘으면 마지막 후보를 제거한다.

이 구조 때문에 조회 시 하위 트리를 다시 순회하지 않아도 된다. 이미 각 접두어 노드가 그 접두어의 인기 후보를 갖고 있기 때문이다.

## 조회 흐름

사용자가 검색어를 입력하면 `suggest(prefix, limit)`가 실행된다.

1. 입력이 비어 있으면 루트 노드의 인기 후보를 사용한다.
2. 입력이 있으면 `normalizer.queryKeys(prefix)`로 조회 키를 만든다.
3. 각 조회 키에 해당하는 트라이 노드를 찾는다.
4. 찾은 노드들의 `topSuggestions`를 합친다.
5. 같은 검색어가 여러 키에서 발견될 수 있으므로 검색어 문자열 기준으로 중복 제거한다.
6. 최종 후보를 다시 인기순으로 정렬하고 `limit`개만 반환한다.

예를 들어 `ㅎㄱ`을 입력하면 초성 색인 키를 통해 `한국 여행`, `한국어 공부`, `한글 입력기` 같은 후보를 찾을 수 있다. `한국여`처럼 공백을 생략한 입력은 compact 키를 통해 `한국 여행`으로 이어진다.

## 순위 정책

순위는 `SuggestionRanking`에서 관리한다.

```java
Comparator.comparingLong(Suggestion::frequency).reversed()
        .thenComparing(Suggestion::query);
```

즉 검색 빈도가 높은 검색어가 먼저 나오고, 빈도가 같으면 검색어 문자열 기준으로 정렬한다. 샘플 데이터는 `SampleSearchTerms`에 고정 목록으로 들어 있으며, 실제 대규모 시스템이라면 이 목록은 로그 집계나 배치 작업 결과로 만들어진 인기 검색어 테이블 또는 트라이 스냅샷으로 대체될 수 있다.

## 라이브러리 방식과의 비교

비교 대상인 `KoreanUtilsAutocompleteEngine`은 전체 검색어 목록을 인기순으로 정렬해 둔 뒤, 요청이 올 때마다 `KoreanUtils.startsWith(term.query(), prefix)`로 필터링한다.

직접 구현 트라이는 색인 시점에 비용을 더 쓰고, 조회 시점에는 접두어 노드의 캐시를 바로 읽는 구조다. 반면 라이브러리 방식은 구현은 단순하지만, 요청마다 검색어 목록을 순회해야 한다.

이 데모에서 두 방식을 같이 둔 이유는 다음 차이를 확인하기 위해서다.

- 직접 구현: 자모 키, 공백 제거 키, 초성 키를 트라이에 미리 색인한다.
- 라이브러리 구현: 한글 prefix 판정은 라이브러리에 맡기고 리스트를 필터링한다.
- 직접 구현: 노드별 top-K 캐시로 조회 비용을 줄인다.
- 라이브러리 구현: 데이터가 커지면 요청마다 순회 비용이 커진다.

## API와 화면

핵심 API는 다음 두 개다.

- `GET /api/autocomplete/compare?prefix=한&limit=5`
  - 직접 구현 트라이 결과
  - `korean-utils` 결과
  - 조회 키, 색인 키 예시, 트라이 노드 수, 검색어 수 반환
- `GET /api/autocomplete/terms`
  - 데모에 사용하는 샘플 검색어 목록 반환

브라우저 화면은 `templates/index.html`에서 구성했고, API 응답을 이용해 두 엔진의 결과와 정규화 정보를 보여준다.

## 테스트한 내용

테스트는 다음 케이스를 중심으로 작성했다.

- 접두어 노드의 top-K 캐시에서 인기순 결과가 반환되는지
- `ㅎㄱ` 같은 초성 접두어로 한글 검색어가 조회되는지
- `한국여`처럼 공백을 생략한 입력이 처리되는지
- `값싼`처럼 복합 받침이 들어간 검색어가 입력 과정에 맞게 처리되는지
- 영문 검색어의 대소문자와 공백 축약이 처리되는지
- `/api/autocomplete/compare` API가 직접 구현 트라이와 라이브러리 결과를 함께 반환하는지

실행은 다음 명령으로 확인한다.

```powershell
cd 13_검색어자동완성시스템/autocomplete-demo
powershell -ExecutionPolicy Bypass -File .\test-demo.ps1
```

## 정리

직접 구현한 핵심은 한글을 트라이에 맞게 그대로 넣는 것이 아니라, 자동완성 입력 과정에서 실제로 들어올 수 있는 형태로 검색 키를 여러 개 만들어 색인한 점이다. 그리고 각 노드에 top-K 후보를 캐시해서 13장에서 설명한 "접두어 노드까지 찾은 뒤 하위 트리를 매번 순회하는 문제"를 줄였다.

현재 구현은 데모 목적이라 샘플 데이터가 메모리에 고정되어 있다. 실제 서비스로 확장한다면 로그 수집, 인기 검색어 집계, 트라이 스냅샷 생성, 캐시 교체, 금칙어 필터링, 국가/언어별 트라이 분리 같은 13장 설계 요소를 붙이면 된다.
