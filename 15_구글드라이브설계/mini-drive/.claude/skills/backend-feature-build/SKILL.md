# Backend Feature Build

## Overview

이 Skill은 백엔드 엔지니어가 한 기능 슬라이스를 **엔티티→리포지토리→서비스→컨트롤러→테스트** 순서로 일관되게 만들도록 돕는다. 기준은 항상 `artifacts/01-architecture/*`이며, 계약에 없는 결정이 필요하면 구현하지 않고 설계자에게 질문한다.

## Workflow

1. 대상 모듈의 계약(엔드포인트·필드·인가·이벤트)을 `api-contract.md`/`db-schema.md`/`websocket-events.md`에서 읽는다.
2. **엔티티/마이그레이션**: JPA 엔티티와 스키마를 계약 DDL과 일치시킨다.
3. **리포지토리**: 조회 인덱스에 맞는 쿼리 메서드를 만든다.
4. **서비스**: 비즈니스 규칙 — 소유자 인가, 파일 상태 전이, 충돌 처리, MinIO(S3 추상화) 입출력, 버전 보존. 실패 시 status=FAILED 및 재시도 경로.
5. **컨트롤러**: 계약의 요청/응답/에러 코드를 그대로 매핑. 인증 필터(JWT) 적용.
6. **실시간**: 해당하면 STOMP 이벤트(FILE_UPLOADED 등)를 발행한다.
7. **테스트**: 정상 + 인가 실패(403) + 토큰 만료(401) + 충돌(409)을 최소 포함. 빌드·테스트 명령 실행.
8. 결과를 `artifacts/02-backend/{module}.md`에 기록하고 qa-reviewer에 완료 메시지를 보낸다.

## 산출물 형식 (`artifacts/02-backend/{module}.md`)

```md
## 모듈: {auth|files|folders|share|versions|trash|sync}
## 구현 범위
- 엔드포인트:
- 엔티티/테이블:
- 이벤트:
## 계약 대조
- [x] 요청/응답 필드 일치
- [x] 인가 규칙 적용
- [x] 상태머신/충돌 처리
## 빌드·테스트
- 명령:
- 결과: 통과 N / 실패 M
## 확인 필요
-
```

## 품질 기준

- 모든 변경/조회 엔드포인트에 소유자 인가 검증이 있다.
- MinIO 접근이 S3 인터페이스로 추상화되어 있다(AWS S3 전환 가능).
- 파일 상태 전이와 충돌(Conflict) 규칙이 계약대로 동작한다.
- 테스트에 정상·인가실패·토큰만료·충돌이 포함된다.
- 빌드·테스트 결과가 로그 파일에 남는다.

## 예외와 금지 행동

- 계약에 없는 엔드포인트·필드를 임의로 추가하지 않는다(설계자에 질문).
- `frontend/`·인프라 설정을 수정하지 않는다.
- `docker compose -v`, DB drop/reset 등 파괴적 명령을 사람 승인 없이 실행하지 않는다.
