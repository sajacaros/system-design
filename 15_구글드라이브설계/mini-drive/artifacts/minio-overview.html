Object Storage · S3 Compatible

MinIO 한눈에 보기

고성능 S3 호환 오브젝트 스토리지. 이 문서는 MinIO의 개념·특징과, Mini Drive 프로젝트에서 실제로 어떻게 쓰고 무엇을 배웠는지를 함께 정리합니다.

개선 요약 — 처음에는 다운로드 시 브라우저에 presigned URL을 직접 발급했는데, 공유 링크를 꺼도 이미 나간 URL이 우회되는 문제가 있었습니다. 이를 nginx 게이트웨이 경유로 바꿔 “공유 링크 비활성화 → 즉시 차단”을 구현했습니다. 그 배경·원리·실전 버그는 6·7장에서 다룹니다.

01 MinIO란?

MinIO는 오픈소스 오브젝트 스토리지 서버입니다. 파일을 디렉터리·블록이 아니라 버킷(bucket) 안의 오브젝트(object) 단위로 저장하며, AWS S3 API와 호환됩니다. 즉 S3용으로 작성한 코드를 거의 그대로 MinIO에 붙일 수 있고, 반대로 MinIO로 개발한 뒤 운영에서 AWS S3로 옮기기도 쉽습니다.

단일 바이너리로 가볍게 띄울 수 있어 로컬 개발·테스트에 특히 좋고, 분산 모드로 묶으면 대용량·고가용 스토리지로 확장됩니다.

메타데이터(파일명·소유자·버전)는 PostgreSQL에, 실제 바이트는 MinIO에 저장. 업로드는 multipart로 백엔드를 경유하고, 다운로드는 nginx 게이트웨이를 경유한다: 백엔드가 매 요청 인가 후 X-Accel-Redirect로 nginx가 MinIO에서 직접 스트리밍(서버 대역폭은 비경유, 7장).

📂 오브젝트 키 설계

minidrive/                  # 버킷
├── users/{userId}/{fileId}  # 현재 원본
├── versions/{fileId}/v1     # 버전 스냅샷
│                 /v2
└── thumbnails/{fileId}.png  # 썸네일

버전을 별도 키로 보존 → 이전 버전 복구 시 유실 0%.

🔧 S3 추상화 (전환 가능성)

스토리지 접근을 StorageService 인터페이스 (put / get / presignGet / delete)로 감싸고, 구현은 AWS SDK v2 S3Client (path-style)로 작성.

MinIO → endpoint/키만 교체 → AWS S3

코드 변경 없이 운영 스토리지를 S3로 전환할 수 있게 설계.

05 버전 관리는 누가 하나?

“버전 관리”에는 두 가지 방식이 있습니다. Mini Drive는 그중 앱 레벨 버저닝을 택했고, MinIO 네이티브 버저닝은 사용하지 않습니다(버킷은 un-versioned). 즉 MinIO 입장에선 서로 다른 객체일 뿐이고, “같은 파일의 버전”이라는 의미는 전적으로 PostgreSQL이 압니다.

채택 ✓

앱 레벨 버저닝

버전마다 다른 키로 저장하고, 이력은 file_version 테이블(DB)이 추적.

미사용

MinIO 네이티브 버저닝

S3 versioning을 켜면 같은 키의 과거 버전을 MinIO가 자동 보관(version-id로 접근).

🗂️ 버전 = 별도 키 (런타임 실제 모습)

users/1/3          # 현재 버전 원본
versions/3/v1      # 버전 1 스냅샷
versions/3/v2      # 버전 2 스냅샷
versions/1/v1,v2,v3 ...

버킷 조회 시 me/minidrive is un-versioned — MinIO 버저닝 OFF.

🧭 DB가 진실의 원천

file_version 테이블이 버전 메타를 보관:

file_id | version | object_key       | size
   3    |    1    | versions/3/v1    |  22
   3    |    2    | versions/3/v2    |  10

users/{userId}/{fileId}는 항상 현재 버전을 가리킴.

⚙️ 동작

  • 새 버전 업로드: versions/{fileId}/v{n}에 저장 + file_version 행 추가, 현재 키 갱신
  • 복구: 과거 v{k}를 새 버전으로 복사 생성 → 이력 보존(유실 0%)
  • 특정 버전 다운로드: 그 버전 키로 게이트웨이 스트리밍(7장)
  • 충돌: baseVersion 불일치 시 409(먼저 저장 우선)

왜 앱 레벨인가 (트레이드오프)

앱 레벨 장점

  • 버전 메타를 DB로 질의·정렬·권한검증
  • 모든 S3 호환 스토리지에서 동일 동작(이식성)
  • 복구/삭제 로직을 완전히 통제

네이티브 버저닝이라면

  • mc version enable 한 줄로 자동 보관
  • 단, DB 이력과 이중 관리·중복
  • 메타 질의가 S3 API에 묶임

운영에서 실수 삭제 보호·백업이 더 필요하면, 네이티브 버저닝을 보조로 켜는 선택지가 있습니다.

06 presigned URL은 어떻게 동작하나

참고 — 아래 SigV4 원리는 그대로 유효하지만, 이 프로젝트에서는 presigned URL을 브라우저에 발급하지 않습니다. nginx↔MinIO 내부 전용(브라우저 비노출)으로만 쓰이고 TTL도 60초로 짧습니다. 브라우저용 다운로드 흐름은 7장의 게이트웨이 모델을 보세요.

핵심은 비밀키를 노출하지 않고도, 특정 요청을 특정 시각까지만 허가했다는 걸 서명으로 증명하는 것입니다. AWS SigV4(Signature Version 4) 방식이며 MinIO도 S3 호환이라 동일하게 씁니다.

비유: 봉인된 결재 도장. 도장(secret key)은 금고에 있고, "○○파일, △시까지 유효"라 적고 서명해 내보낸다. 받는 쪽은 도장 없이 서류만 내면 되고, 내용을 고치면 도장 자국과 안 맞아 들통난다.

🔬 실제 발급 URL 해부

http://minio:9000/minidrive/users/1/3   # 내부 endpoint(브라우저 비노출)
  ?X-Amz-Algorithm=AWS4-HMAC-SHA256   # 서명 알고리즘
  &X-Amz-Credential=minioadmin/2026.../s3/aws4_request # 누구+범위
  &X-Amz-Date=20260617T011500Z        # 서명 생성 시각
  &X-Amz-Expires=60                   # 유효 기간(초) — 내부 전용 60초
  &X-Amz-SignedHeaders=host           # 서명에 포함된 헤더(host=minio:9000)
  &X-Amz-Signature=2091a6c649...      # HMAC 서명 결과

secret key 자체는 URL에 없습니다. 들어가는 건 access key 식별자와 일방향 해시(서명)뿐. 이 URL은 백엔드가 X-Accel-Redirect로 nginx에만 넘기며 브라우저로는 흐르지 않습니다.

① 서명 생성 — 백엔드

  1. 요청 정규화: 메서드+경로+쿼리(expires·date 포함)+헤더
  2. 정해진 규칙으로 "string-to-sign" 구성
  3. secret key로 파생키 → HMAC-SHA256 계산 = 서명
  4. 서명을 URL 쿼리에 붙여 완성

② 검증 — MinIO

  1. 들어온 요청에서 경로·쿼리·헤더·시각을 꺼냄
  2. 자기 secret key로 같은 계산을 다시 수행
  3. 재계산 서명 == URL의 서명 → 허가
  4. 하나라도 다르면 거부(SignatureDoesNotMatch)

⏱️ "기간(expires)"이 위조 불가능한 이유

  • X-Amz-Date + X-Amz-Expires는 서명 계산의 입력값 → 기간을 늘리면 서명이 안 맞아 무효
  • 기간을 정당히 바꾸려면 secret key가 필요한데, 클라이언트엔 없음
  • MinIO는 추가로 현재 시각 > (date+expires) 이면 거부(Request has expired)

→ 서명으로 잠그고 + 시각 검사로 강제하는 이중 장치.

⚠️ bearer 특성 — 개별 취소 불가: 발급된 presigned URL은 만료 전까지 URL을 가진 누구나 접근 가능하고, S3/MinIO 구조상 개별 무효화가 불가능합니다(MinIO GitHub #5180·#14491 등 “working as intended”). 이 때문에 공유 링크를 꺼도 이미 발급된 URL이 TTL 동안 우회되는 문제가 있었고 → 브라우저에 presign을 주지 않는 게이트웨이 모델로 전환해 해결했습니다(바로 아래 7장).

07 실전에서 부딪힌 교훈

핵심

공유 링크를 꺼도 MinIO로 직접 다운로드되던 문제

공유 링크를 비활성화(is_active=false)해도, 비활성화 직전에 발급된 presigned URL이 TTL(당시 5분) 동안 살아 있어 MinIO에 직접 접근하는 우회가 가능했습니다. presigned URL은 S3/MinIO 구조상 개별 취소가 불가능합니다(MinIO GitHub #5180 등 “working as intended”).

이전 방식 — 우회 가능

브라우저 →(5분 presigned URL)→ MinIO 직접. MinIO는 is_active를 모르므로 서명만 유효하면 내줌 → 비활성화 우회.

개선 후 — 즉시 차단

브라우저 → nginx → 백엔드(매 요청 인가) → X-Accel-Redirect → nginx가 MinIO에서 스트리밍. 끄면 다음 요청부터 즉시 410.

브라우저 → nginx → 백엔드 : is_active/만료/소유자 실시간 인가
                        ├─ 거부 → 410 DISABLED (바이트 안 나감)
                        └─ 허가 → X-Accel-Redirect: /_minio/... (바디 없음)
            nginx ──internal──→ MinIO(minio:9000) : 스트리밍 (백엔드 비경유)

내부 presign(60초)은 nginx만 보고 브라우저엔 노출되지 않습니다 — 더 이상 “보안 경계”가 아니라 nginx가 MinIO 문을 여는 일회용 열쇠일 뿐. 보안 판정은 그 앞 백엔드 인가에서 끝납니다. MinIO API 포트(9000)도 호스트에서 비노출해 직접 접근 경로 자체를 제거했습니다.

🧪 실기동 E2E에서만 드러난 nginx 함정 3종

정적 검증·단위테스트는 모두 통과했지만, 실제 컨테이너로 띄우니 nginx↔MinIO 연동에서 셋이 터졌습니다.

  • Host 헤더가 서명과 어긋남 — upstream minidrive_minio처럼 named upstream으로 프록시하면 nginx가 보내는 기본 Host($proxy_host)가 upstream 이름(minidrive_minio, 언더스코어 포함)으로 나갑니다. 그러면 두 가지가 동시에 깨집니다: ① MinIO가 언더스코어 호스트를 invalid hostname으로 거부, ② 백엔드가 minio:9000 기준으로 만든 SigV4 서명의 host와 불일치해 SignatureDoesNotMatch.
    → 해결: proxy_set_header Host minio:9000; — 서명에 쓰인 값과 글자 그대로 같은 Host를 강제해 서명을 일치시킴.
  • Authorization 헤더 누수 — 인증 다운로드(/api/files/…)에서는 브라우저가 Authorization: Bearer <JWT>를 함께 보냅니다. 이 헤더를 nginx가 MinIO까지 그대로 전달하면 URL 쿼리의 presign(SigV4)과 인증 수단이 두 개가 되어 MinIO가 multiple authentication types로 거부합니다. MinIO 인증은 오직 쿼리 presign으로만 합니다.
    → 해결: proxy_set_header Authorization ""; — MinIO로 가는 인증 헤더를 빈 값으로 덮어 presign만 남김.
  • 단일 파일 bind-mount의 inode 교체 — nginx.conf를 파일 하나로 마운트하면, 에디터가 atomic-save(임시 파일에 쓴 뒤 rename)로 저장할 때 inode가 바뀝니다. 컨테이너는 옛 inode를 계속 들고 있어, 내용을 고치고 nginx -s reload를 해도 변경이 반영되지 않습니다(reload 무효). "고쳤는데 왜 안 바뀌지"로 한참 헤맬 수 있습니다.
    → 해결: 설정 변경 후 docker compose up -d --force-recreate로 컨테이너 재생성(혹은 파일 대신 디렉터리를 마운트해 inode 교체 자체를 회피).
지금은 해소됨

presigned URL의 “호스트 주소” 함정

예전엔 컨테이너 내부 주소(http://minio:9000)로 presign을 발급하면 브라우저가 minio 호스트를 몰라 다운로드가 실패했고, 그래서 MINIO_PUBLIC_ENDPOINT(http://localhost:9000)를 따로 두어 presign 발급에만 공개 호스트를 썼습니다.

게이트웨이 모델로 바꾸면서 이 문제가 통째로 사라졌습니다 — 브라우저가 더 이상 presign을 받지 않으므로 MINIO_PUBLIC_ENDPOINT를 제거했고, presign은 내부 endpoint(minio:9000)로만 서명합니다.

그 외: 기동 시 mc 일회성 컨테이너로 minidrive 버킷 자동 생성, 헬스체크로 기동 순서 보장.
